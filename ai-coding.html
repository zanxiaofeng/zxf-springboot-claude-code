执行摘要

AI编码代理的上下文管理已从简单的"窗口填充"演进为精密的系统工程。核心机制包括：基于代理循环的三阶段工作流（收集-执行-验证）实现动态决策；通过三级渐进压缩（工具修剪→缓存优化→结构化摘要）应对200K token硬约束与软退化；技能系统作为动态上下文修饰器按需注入领域知识；以及静态（CLAUDE.md）与动态（运行时状态）上下文的协同管理。Claude Code与Codex在压缩哲学上形成鲜明对比：前者追求"精准遗忘"与缓存友好，后者采用"交接备忘录"式全量摘要。当前趋势正从提示工程转向上下文工程，强调"加载什么"比"说什么"更重要。

技术突破

  • • 代理循环架构实现迭代式决策调整
  • • 九部分结构化摘要防止信息漂移
  • • 工具搜索工具减少85%上下文占用
  • • 会话记忆路径实现即时压缩

应用价值

  • • 长会话性能稳定性提升
  • • 多模型协作优化资源利用
  • • 技能系统实现领域专业化
  • • 上下文工程降低使用成本

1. 核心架构与信息整合决策流程

1.1 代理循环（Agentic Loop）机制

三阶段工作流

AI编码代理的核心运作范式建立在持续运转的代理循环之上。以Claude Code为例，其系统被明确设计为分层代理系统，而非单一助手通过超长提示进行对话的简化模型[1]。

上下文收集

从九个有序来源构建完整上下文窗口，包括系统提示、环境信息、CLAUDE.md指令等

行动执行

工具调用的并行化与序列化智能调度，采用StreamingToolExecutor降低延迟

结果验证

通过消息链注入、检查点机制和反应式压缩实现反馈闭环

迭代式决策机制

代理循环的迭代特性体现在每一步决策都严格依赖前序步骤的输出状态。当模型读取文件后发现新的依赖关系，可能触发额外的文件读取；执行测试失败后，可能转向调试或代码修复。这种动态调整机制使得代理能够在复杂任务中逐步推进，而非试图一次性解决所有问题[2]。

1.2 多源信息融合框架

用户输入解析

Claude Code采用"目标-约束-完成标准"（Goal-Constraint-Done）框架对任务进行结构化解析，通过查询增强策略将非结构化意图转化为结构化任务描述。

示例：从"添加邮件通知"到明确的目标、约束和完成标准定义

任务状态追踪

通过多层次的待办事项系统维护精确的任务状态表示，包括用户可见的TODO列表、模型内部的计划状态、以及压缩摘要中的结构化任务追踪。

机制：九部分结构化摘要专门用于任务状态持久化

1.3 上下文生成与决策输出

行动选择机制

在每一轮代理循环中，模型面临的核心决策是：继续执行工具调用以获取更多环境信息，还是基于已有信息生成文本回复。这一选择涉及对多种因素的综合权衡：当前上下文中信息的充分性、信息的不确定性、任务的紧急程度、以及工具调用的成本效益分析。

Claude Code的决策哲学

采用"拒绝优先评估"（deny-first evaluation）——系统默认对潜在风险操作持保守态度，通过权限模式、ML分类器和可选的沙箱执行等多层机制逐步建立信任。

Codex CLI的自主策略

系统提示明确编码了"自主性高级工程师"的角色期望："一旦用户给出方向，主动收集上下文、计划、实施、测试和完善"。

2. 上下文管理与优化：渐进式压缩体系

2.1 上下文作为稀缺资源的设计哲学

硬约束与软退化

上下文窗口是AI编码代理面临的最根本约束之一。Claude Code基于Claude 3.5 Sonnet等模型，标准上下文上限为200K token，但实用上限因系统开销而大幅缩减。更为关键的是"软退化"现象：当上下文填充超过20-40%时，模型输出质量开始下降；超过60%时，多数从业者观察到明显的性能衰减[3]。

2.2 Claude Code三级"精准遗忘"机制

L1 第一层：工具结果修剪

完全由本地规则引擎驱动，不涉及任何LLM API调用，具有零额外成本和即时执行的特性。识别并保护最近工具调用的结果，将超出保护窗口的旧工具结果替换为标准化占位符。

[Old tool result content cleared]

L2 第二层：缓存友好策略

对Anthropic API Prompt Cache特性的深度优化。采取"保护前缀、修剪尾部"的策略，确保消息序列的前缀部分与前一请求保持绝对一致，从而使API缓存命中。

经济效益：缓存优化可将长时任务的总成本降低一个数量级，同时显著改善响应延迟

L3 第三层：九部分结构化LLM摘要

基于LLM的结构化摘要，采用九部分精细模板，强制在特定信息槽位中填充内容，防止关键信息的系统性遗漏。包括用户原始意图、核心技术概念、关键文件与代码等[4]。

2.3 Codex CLI的"交接备忘录"机制

核心机制

保留所有用户消息的原文，但删除所有assistant回复和工具相关消息，然后插入一个包含结构化摘要的fabricated assistant消息。这种"用户消息保护"策略体现了对"用户意图神圣性"的尊重。

优势：直观性强，"工作交接"是人类开发者普遍理解的概念

潜在风险

"全有或全无"的特性意味着如果摘要遗漏关键细节，该信息将永久丢失，无法通过任何机制恢复。社区开发者通过扩展压缩提示增强保真度[5]。

改进：10条规则增强压缩保真度，显著改善多次压缩后的信息保留
高级压缩形式

通过 /responses/compact端点实现"加密内容"压缩，优化重建保真度而非人类可读性，压缩率高达99.3%。这种"潜在空间压缩"方法可能保留比显式摘要更多的信息，特别是那些难以用语言表达的微妙模式。

3. 技能与工具的动态调用体系

3.1 技能系统（Skills System）架构

声明式发现与调用机制

Claude Code的技能系统采用独特的声明式架构，将技能视为"动态上下文修饰器"而非"可执行代码"。技能以Markdown文件存在，包含YAML frontmatter和指令主体，通过两阶段处理实现高效的按需加载[6]。

阶段一：索引构建

会话启动时扫描技能目录，仅解析YAML frontmatter构建轻量级索引

阶段二：自动匹配

基于用户意图与技能描述的语义匹配，决定是否调用技能

阶段三：动态注入

技能内容注入上下文，修改代理的行为约束和知识基础

关键优势

决策的完全内生性避免了传统意图分类系统的脆弱性，同时利用大型语言模型强大的语义匹配能力。技能发现遵循"Claude搜索优化"约定——每个技能的描述以"Use when..."开头。

技能作为动态上下文修饰器

技能的本质是"专业化提示模板"，它们在调用时动态修改代理的上下文环境，而非执行预定义的操作序列。这种设计使技能具有极高的灵活性——同一个技能可以被不同模型版本以不同方式解释和执行。

上下文修饰能力
  • • 修改工具权限
  • • 覆盖模型参数
  • • 注入生命周期钩子
  • • 调整思考深度
三级优先级覆盖
  • • 项目本地技能（最高）
  • • 用户全局技能
  • • 内置技能（最低）

3.2 外部工具的智能加载与发现

工具搜索工具

按需发现替代全量加载，允许代理在需要时搜索可用工具而非预先加载全部定义。这种设计将工具发现从"广播"模式转变为"查询"模式[7]。

MCP动态加载

遵循Model Context Protocol标准，实现跨代理兼容性和显著的上下文节省。LinkedIn的CAPT框架通过MCP为工程师提供对内部系统的标准化访问。

精度提升：Opus 4从49%提升至74%，Opus 4.5从79.5%提升至88.1%

结果整合机制

工具执行结果的上下文整合遵循严格的消息链格式：assistant → tool_use → user → tool_result，形成清晰的调用-响应配对。对于超大规模输出，采用智能截断策略——超过10,000字符的结果保存为临时文件，上下文中仅保留预览片段与文件路径。

4. 特定类型上下文的创建、管理与使用

4.1 静态上下文：CLAUDE.md与项目规范

四级层次结构

CLAUDE.md采用四级层次结构设计，允许规范的分层继承和覆盖。项目级CLAUDE.md存储团队共享约定，而CLAUDE.local.md容纳个人偏好，避免污染版本控制[8]。

Managed层级

/etc/claude/ - 系统级，企业安全策略

User层级

~/.claude/ - 用户级，个人编码风格

Project层级

./CLAUDE.md - 项目级，技术栈和架构决策

Local层级

./CLAUDE.local.md - 本地级，个人工作流

跨会话不变性特权

CLAUDE.md享有特殊的上下文地位：在自动压缩和手动/clear操作中，其内容被重新加载，确保跨会话的规范一致性。这种"免疫特权"基于设计意图——项目规范不应随对话历史被压缩丢弃。

物理分离机制

存储于磁盘，每次请求时重新读取，而非保存在可变上下文中

即时生效特性

内容变更立即生效，无需重启会话，但需保持文件精简

4.2 动态上下文：运行时信息与状态

代码库实时状态

Claude Code作为终端原生代理，直接读取本地文件系统，获取当前工作目录、文件树结构、最近访问文件、Git状态等信息，实现真正的结对编程体验。

自动重读机制

压缩后读取最多5个最近编辑文件，总预算50K tokens，确保掌握最新内容

Hooks动态注入

在特定生命周期事件点执行自定义脚本，通过additionalContext和systemMessage输出动态扩展代理上下文。

关键钩子类型
  • • PreToolUse - 安全警告、合规检查
  • • PostToolUse - 结果分析、自动重试
  • • UserPromptSubmit - 查询增强、意图识别

条件上下文加载

多项目工作流中的动态上下文切换：基于当前工作目录、最近访问文件、用户显式指示自动检测活跃项目，切换时卸载前一项目上下文、加载新项目上下文、保留全局上下文。

路径模式匹配

.claude/rules/*.md文件支持按文件路径模式匹配激活，例如仅在编辑server/api/目录下的文件时加载后端特定规则。

Active Context = Global CLAUDE.md + Current Project CLAUDE.md

4.3 记忆层次与持久化机制

三层记忆架构

用户级记忆

~/.claude/ - 全局跨项目，通过"记忆合成"自动处理对话，每约24小时提取长期价值信息。

典型内容：职业、语言偏好、常用工具、沟通风格
项目级记忆

.claude/memory/ - 项目隔离，每个Project拥有独立的隔离记忆空间，偏好和上下文仅适用于该Project内的对话。

典型内容：项目特定规范、技术栈、团队约定、调试洞察
动态导入级

5. 趋势与未来展望

从提示工程到上下文工程的范式转移

2026年的核心趋势表明，AI编码代理的优化重点已从"说什么"转向"加载什么"。系统提示、文件、记忆、示例的结构比任务措辞本身更重要。这一转变反映了上下文管理从战术技巧向战略架构的演进。

技能系统

封装可复用的领域知识，按需加载避免前置膨胀

上下文工程

优化信息组织与压缩，最大化信号密度，最小化噪声

工具设计

确保高效的结果格式，降低处理负担，提高决策精度

子代理配置

实现任务分解与隔离，控制单窗口复杂度

任务分块策略与80/20规则

HumanLayer团队提出的"频繁有意压缩"（Frequent Intentional Compaction）工作流将上下文管理融入整个开发流程，保持利用率在40%-60%范围。这种"分而治之"策略使得代理能够在有限上下文内处理远超窗口大小的复杂项目。

研究阶段

理解代码库，定位相关组件，创建SPEC.md规范文档。广泛收集，容忍噪音。

输出产物：SPEC.md - 结构化研究文件

计划阶段

构建变更路线图，明确测试策略，生成PLAN.md实施计划。生成可审查的实施计划。

输出产物：PLAN.md - 包含文件变更、代码片段和测试策略

实施阶段

编写和审查代码，保持窗口效率<40%，标记完成任务，聚焦剩余工作。

关键指标：上下文利用率建议低于40%容量

未来发展方向

多代理协作

Agent Teams模式实现"大脑-双手"分离，Team Lead使用Opus规划，Teammate使用Sonnet执行，进一步优化成本与效率平衡。

潜在空间压缩

加密内容压缩优化重建保真度而非人类可读性，可能保留更多难以用语言表达的微妙模式。

自适应上下文

基于任务类型感知的动态阈值，为不同复杂度操作自动调整上下文管理策略。

本研究基于公开技术文档和社区实践，深入分析了AI编码代理动态上下文组织与生成机制的核心原理与发展趋势。
